# Project status (2026-01-04)

## Status
- Soft wordlist constraint mode is implemented and working end-to-end.
- Constraints are still computed via `lm-format-enforcer`, but enforcement can be either:
  - **Hard**: disallowed tokens are masked out via `prefix_allowed_tokens_fn` (original behavior)
  - **Soft**: disallowed tokens remain possible, but receive a configurable logit penalty via a custom `LogitsProcessor`

## Progress / what changed
- Added `vocab_constraint_mode` and `vocab_soft_penalty` knobs:
  - Environment defaults: `VOCAB_CONSTRAINT_MODE` (default `hard`), `VOCAB_SOFT_PENALTY` (default `8.0`)
  - Per-request override on `/v1/chat/completions` and batch jobs
- Implemented `SoftPrefixConstraintLogitsProcessor` to penalize (not mask) tokens outside the lm-format-enforcer allowed set.
- Added `scripts/trace_vocab_penalty.py` to debug/trace how the penalty interacts with Transformers v5 generation (including beam sampling) without reading large repo wordlists.

## Key findings from tracing
- With `num_beams > 1` + `do_sample=True` (beam sampling path), Transformers v5 applies sampling controls (temperature/top-k/top-p) as part of the processor chain.
- The observed “cliff” where the soft penalty appears to become a hard ban is typically caused by **hard truncation** from `top_k` / `top_p` (tokens falling out of the candidate set after penalization).

## Next steps
1. Implement rank-aware soft penalties (requested): penalize tokens more strongly when they can only lead to words ranked beyond `vocab_n_words`.
   - Start with a low-risk tiered approach (multiple cached cutoffs: `N`, `2N`, `4N`, ...) and map tiers to penalty multipliers.
2. Add configuration for tier strategy and multipliers (env + request fields), and document defaults in code.
3. Extend the trace script (or add a small benchmark mode) to measure CPU overhead per step/beam for tiered penalties.
4. Run validators after implementation changes.
